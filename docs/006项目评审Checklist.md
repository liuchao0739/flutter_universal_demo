## 一、项目基础合规性

### 项目结构

* [ ] 是否使用公司标准 mono-repo 结构
* [ ] apps / packages / docs 是否分离
* [ ] 是否存在业务代码直接写在 app 层

---

## 二、架构与分层

### 页面层（Page）

* [ ] Page 是否只负责 UI 和交互
* [ ] Page 是否直接调用 SDK（❌ 不允许）
* [ ] 是否通过 Router 统一跳转

### Service 层

* [ ] 是否所有 SDK 都有 Service 封装
* [ ] Service 是否无 UI 依赖
* [ ] 是否支持 Mock / 替换

---

## 三、状态与数据流

* [ ] 是否统一使用 Riverpod
* [ ] 是否存在 setState 滥用
* [ ] 状态是否可追踪、可测试

---

## 四、网络与数据

* [ ] 是否通过 ApiService 发起请求
* [ ] 是否有 Mock 数据
* [ ] 是否有接口错误兜底

---

## 五、Crash / 日志

* [ ] 是否接入 CrashService
* [ ] 是否能手动触发 Crash 测试
* [ ] 是否记录页面 PV / UV

---

## 六、权限与合规

* [ ] 权限是否有 UI 触发
* [ ] Android Manifest 是否最小化
* [ ] iOS Info.plist 文案是否合规

---

## 七、UI 与体验

* [ ] 是否支持深色 / 浅色模式
* [ ] 是否支持多语言
* [ ] 是否适配主流屏幕尺寸

---

## 八、代码质量

* [ ] 单文件是否 < 300 行
* [ ] 命名是否规范
* [ ] 是否存在重复代码

---

## 九、Demo 与可验证性（重点）

* [ ] 每个核心能力是否有 Demo 页面
* [ ] 打开 App 是否能直接体验主要功能
* [ ] Demo 是否与 Service 解耦

---

## 十、可扩展性评估

* [ ] 新增模块是否低侵入
* [ ] 是否能被其他 App 复用
* [ ] 是否满足未来 6–12 个月扩展需求

---

# 结论（非常重要）

> 如果一个 Flutter 项目：

* 通过了这份 Checklist
* 符合白皮书架构
* Demo 可完整跑通

那么它已经具备 **企业级上线与长期维护能力**。
