---

## 1️⃣ KPI 设计原则

* 不唯代码量
* 强调质量与影响力
* 鼓励平台化贡献

---

## 2️⃣ KPI 分层模型

```
个人 KPI
   ↓
项目 KPI
   ↓
平台 KPI
```

---

## 3️⃣ 个人技术 KPI（示例）

| 维度 | 指标      |
| -- | ------- |
| 架构 | 是否遵循规范  |
| 交付 | 准时率     |
| 质量 | Crash 率 |
| 贡献 | 中台模块    |

### 示例评分

* 新增 Service：+10
* Demo 可跑：+5
* 引入技术债：-10

---

## 4️⃣ 项目级 KPI

| 指标      | 目标     |
| ------- | ------ |
| Crash 率 | < 0.1% |
| 页面性能    | < 16ms |
| 代码复用率   | > 70%  |
| 评审通过率   | 100%   |

---

## 5️⃣ 平台级 KPI（最重要）

| 指标       | 说明    |
| -------- | ----- |
| 新项目启动时间  | ≤ 1 天 |
| SDK 接入重复 | 0     |
| 模块复用数    | 持续增长  |
| 业务线满意度   | ≥ 90% |

---

## 6️⃣ KPI 与晋升挂钩建议

* 核心 Service Owner：优先晋升
* 中台模块维护者：长期加分
* 破坏规范者：扣分
