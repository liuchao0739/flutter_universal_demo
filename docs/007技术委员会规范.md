# 一、《Flutter 技术委员会规范》

**文件名建议**：
`docs/FLUTTER_TECH_COMMITTEE_GUIDE.md`

---

## 1️⃣ 技术委员会定位

### 1.1 委员会职责

Flutter 技术委员会是公司 Flutter 技术体系的 **最高技术决策机构**，职责包括：

* Flutter 技术路线决策
* 架构规范制定与演进
* 核心模板与基础库治理
* 技术风险评估与兜底
* 跨项目技术一致性保障

---

## 2️⃣ 组织结构

### 2.1 角色定义

| 角色   | 职责        |
| ---- | --------- |
| 主任委员 | 技术路线最终裁决  |
| 架构委员 | 架构设计与演进   |
| 平台委员 | 工程化 / 自动化 |
| 业务委员 | 业务诉求反馈    |
| 秘书   | 会议纪要 / 推进 |

---

## 3️⃣ 决策范围（边界）

### 委员会 **必须介入** 的事项：

* Flutter 大版本升级
* 核心架构调整
* 状态管理 / 路由 / 网络方案变更
* 新 SDK（支付 / 地图 / IM）引入
* mono-repo 结构调整

### 委员会 **不介入** 的事项：

* 页面 UI 风格
* 具体业务逻辑
* 单项目局部优化

---

## 4️⃣ 技术提案机制（RFC）

### 4.1 RFC 流程

```
提出 RFC
   ↓
技术委员会评审
   ↓
Demo / PoC 验证
   ↓
决策（通过 / 驳回 / 延期）
   ↓
落地与推广
```

### 4.2 RFC 必须包含

* 背景 & 痛点
* 技术方案对比
* 风险评估
* Demo 或验证结论

---

## 5️⃣ 技术资产治理

* 核心模板必须由委员会维护
* Service / Feature 禁止私有分叉
* 所有基础能力必须有 Demo 页面

---

## 6️⃣ 技术债治理

* 每季度技术债盘点
* 高风险模块强制重构
* 禁止“业务绕过基础能力”
